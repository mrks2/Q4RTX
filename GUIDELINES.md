# Guidelines

Notes on how the code is organized and conventions to follow.

## Proxy vs renderer

The proxy runs inside Quake 4 (32-bit). Its job is to capture GL state and geometry, then shove it into shared memory. That's it — no rendering logic, no D3D12, no scene graph. Crashes here crash the game, so keep it defensive.

The renderer is a standalone 64-bit process. It's class-based with each class owning one thing:

- **D3D12Context** — device, swap chain, command list, fence
- **RTPipeline** — root signature, PSO, shader table, output UAV
- **Scene** — owns Objects, Lights, Skybox, builds acceleration structures
- **Object** — single mesh with vertices, indices, bounds
- **Light** — point light properties
- **Skybox** — procedural sky params as a constant buffer
- **Renderer** — wires everything together, dispatches rays, presents
- **FrameReceiver** — reads frames from shared memory
- **Window** — Win32 window + message pump

Proxy code is in `src/proxy/`, renderer in `src/renderer/`, shared types in `src/shared/`. The proxy should never include anything from the renderer.

## Adding new things to the scene

1. Make a new class with its own `.h/.cpp` in `src/renderer/`
2. Give it `Upload(ID3D12Device*)` if it needs GPU resources
3. Add it as a member of `Scene`
4. If the shader needs data, add a root parameter in `RTPipeline::CreateRootSignature` and the matching buffer in `rt_shaders.hlsl`
5. Add the `.cpp` to `CMakeLists.txt`

## Shared memory rules

The event sync is critical: if you wait on `DataReady` and get it, you *must* signal `DataRead` afterward, even if you throw the frame away. Skipping this deadlocks the proxy.

Shared structs in `shared_memory.h` must be plain C — no vtables, no STL containers, no pointers. Keep them aligned since one side is 32-bit and the other is 64-bit.

## D3D12 / DXR

- `ComPtr<>` for everything, no manual `Release()`.
- Use helpers in `dxr_helpers.h` for buffer creation.
- Upload heaps are fine for per-frame data (geometry, constants). Don't over-optimize with staging.
- Root signature is in `RTPipeline::CreateRootSignature`. Keep the descriptor table for heap-bound resources, use inline CBVs for small constant buffers.
- Single BLAS for all geometry, single TLAS instance with identity transform. Always `PREFER_FAST_BUILD`.
- Shader table has 3 records (RayGen, Miss, HitGroup). If you add more, update `D3D12_DISPATCH_RAYS_DESC` in `Renderer::DispatchRays`.

## Code style

C++20. One class per file pair. `#pragma once`. Member vars prefixed `m_`. Braces on same line. `printf` for logging (goes to `q4rtx_renderer.log`). No exceptions — check HRESULTs and return `bool`. Prefer `uint32_t`/`float` over Windows typedefs except in D3D12 struct init.

HLSL: `float3`/`float4` (no `half`). Entry points and helpers are PascalCase. Constant buffer fields prefixed `g_`. Next free registers: `t4`, `b2`.

## Build notes

Both targets share one `CMakeLists.txt`. Proxy builds as Win32, renderer as x64. Don't edit `gl_passthrough.cpp` — it's generated by `gen_passthrough.py`. Shaders are compiled by DXC automatically if available, otherwise run `dxc -T lib_6_3 -Fo shaders/rt_shaders.cso shaders/rt_shaders.hlsl`.
